# Markdown → 公众号推文

## Task

她的 Word 稿是给学界看的，**公众号版是给公众看的**——要求不一样：

| 维度 | Word 版 | 公众号版 |
|------|--------|---------|
| 字体 | 宋体 | 无衬线（微信默认或手动设） |
| 繁简 | 保留繁（学术版本） | **简化**（读者阅读友好） |
| 脚注 | 页底编号 | 正文后直接接小字注释 |
| 引文 | 无特殊样式 | **引用框** |
| 人名地名 | 首次出现加括号注 | 首次出现加下划线 + 生卒年 |
| 图片 | 正文流式 | 满宽 + 题注加灰底卡片 |
| 长度 | 整篇 | 公众号阅读 2000-4000 字是友好区间，过长建议拆 |

输出：

```
<input>.mp.html    ← 带内联 CSS，粘贴到公众号后台 / 直接上传
<input>.mp.md      ← 秀米兼容 Markdown（她想进一步精修时用）
```

## Process

### Step 1：确认输入和长度

```bash
INPUT="<markdown-path>"
test -f "$INPUT" || { echo "文件不存在"; exit 1; }
WC=$(wc -m < "$INPUT")
echo "字数（字符）: $WC"
```

如果 `WC > 5000` → 建议她拆分：

> 这篇 5000+ 字，公众号读完要 15 分钟以上。建议拆成 2-3 篇推送。可以按章节切，还是我帮你按话题聚类？

她说不拆就继续。

### Step 2：繁简处理

历史论文常见繁体字、异体字。公众号读者大部分读简体更顺。三个分支：

| 场景 | 处理 |
|------|------|
| 纯学术古籍研究（保留繁体是学术声明）| 不简化，原样输出 |
| 现代简体论文 | 不处理 |
| 民国排印 / 偶有繁体 / 引用古籍片段 | **分级简化**——主体简化，引文保持原貌 |

命令行 `--simplify` 强制简化全部；无参数时按上表判断。

简化用 OpenCC（zh-hant → zh-hans），脚本里调用。**保护引文**：被「」""或 > 引用的段落不简化，保留原貌（学术引文要忠实）。

### Step 3：调排版脚本

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/mp-format/scripts/md_to_wechat.py" \
    --input "$INPUT" \
    --output "${INPUT%.md}.mp.html" \
    --also-markdown "${INPUT%.md}.mp.md" \
    $( [ "$SIMPLIFY" = "1" ] && echo "--simplify" ) \
    --byline "$BYLINE" \
    --source "$SOURCE"
```

脚本产出 HTML 特性：

1. **全内联 CSS**（公众号后台剥离外联 style）
2. **标题层级**：
   - `#` → 22 pt 加粗 深灰 上下留白
   - `##` → 18 pt 加粗 左侧彩色竖条
   - `###` → 16 pt 加粗
3. **引用块** `>`：灰底 + 左竖条 + 字号小一点，古籍原文引用显得庄重
4. **脚注**：`[^1]` 转成 `<sup>1</sup>`，文末有脚注列表（而非 Word 的页底）
5. **段首缩进**：2 字符（公众号移动端阅读友好）
6. **图片**：满宽 + 底部题注卡片
7. **作者栏**：首行生成 「文 / XXX · 来自 XX 期刊」卡片（byline + source）
8. **封面提示**：告诉她公众号封面图要另配（16:9 建议）

### Step 4：人名地名增强（可选）

如果用户希望**读者友好度更高**，让脚本给首次出现的人名加生卒年注：

- 输入：校对过的 Markdown，包含人名
- 脚本检查 `${CLAUDE_PLUGIN_ROOT}/skills/mp-format/references/people.json`（小型人名库，可扩展）
- 首次出现「陈寅恪」自动变 「**陈寅恪（1890-1969）**」
- 库里没有的人名不动

她可以说"加生卒年"或"不要加注"触发/关闭。

### Step 5：预览

```bash
open "${INPUT%.md}.mp.html"
```

在浏览器里能看到真实的公众号样式预览——排版、字号、间距都是最终效果（公众号会加载相同的内联样式）。

她可以：
1. 肉眼过一遍
2. 直接复制 HTML 源码 → 粘贴到公众号后台「html 模式」
3. 或者复制渲染后的内容 → 粘贴到公众号编辑器（富文本模式保留大部分样式）
4. 或导入秀米做更精致的模板（用 `mp.md` 文件）

### Step 6：报告

```
公众号版已生成：
- HTML（粘贴到后台）：${INPUT%.md}.mp.html
- 秀米 Markdown：${INPUT%.md}.mp.md
- 字数：NNN（简化后）
- 预估阅读时间：X 分钟
- 图片：M 张（封面图需另配）

建议：
1. 在浏览器里看一眼预览，确认无误
2. 粘贴到公众号后台时选 "html 模式"（设置→群发→模板 下方）
3. 或用秀米: 粘贴 mp.md 后选「公众号素材」类模板
4. 封面图：建议用文章主题相关的历史图（避免侵权，国图公众号推送的封面常用公有领域老照片）
```

## 判断规则

- **原文有拉丁字母（外文书名、外国人名）**：HTML 里自动 `<span lang="en">` 包起来，避免公众号字体渲染问题
- **原文有数学公式**：公众号不原生支持 LaTeX。脚本会转成图片（PIL 渲染），但质量有限。告诉用户："公式较多的论文，公众号推送效果有限，建议主打 Word 版。"
- **原文有竖排古籍原文引用**：公众号不支持竖排。脚本转成横排，在引用块上方加小字说明「（原文为竖排）」

## 失败兜底

- OpenCC 没装 → `pip3 install opencc-python-reimplemented`
- 图片路径在 Markdown 里是相对路径找不到 → 脚本会警告并跳过，她需要手动在公众号后台补图
- 粘贴到公众号后样式丢失 → 是富文本模式剥离了，切到 html 模式粘贴

## 跟 to-docx 的关系

她的标准流程：

1. 先跑 `to-docx` 出 Word 稿（学界用）
2. 再跑 `mp-format` 出公众号（给读者）

两者都基于同一份校对定稿 `final.md`，互不干扰。公众号版的简化和增强**不会**污染 Word 版。
